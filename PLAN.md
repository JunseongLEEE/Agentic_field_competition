# SNU AI Challenge — 프레임 순서 예측 대회 실행 계획

> 작성일: 2026-07-04 · 예선 마감: 2026-07-24 (약 20일 남음) · 제출 한도: 1일 2회

## 1. 과제 요약

- **입력**: 뒤섞인 비디오 프레임 4장 + 비디오를 시간 순으로 묘사한 영어 캡션 1문장
- **출력**: 각 입력 프레임의 실제 시간상 위치 `[n, n, n, n]` (4! = 24개 순열 중 하나)
- **평가**: Exact Match Accuracy (완전 일치만 정답, 부분 점수 없음)
- **랜덤 베이스라인**: 1/24 ≈ 4.2% / 항상 `[1,2,3,4]` 제출 시 ≈ 15% 추정 (아래 §2 참고)

### 핵심 제약 (규칙 요약)
| 항목 | 내용 |
|---|---|
| 추론 환경 | **RTX 3090 24GB 1장**, 인터넷 차단, 테스트 전체 24시간 이내 (819샘플 → 샘플당 ~105초, 매우 여유) |
| 외부 데이터 | **학습 사용 금지** (제공된 train만 사용) |
| 오픈소스 모델 | 허용 (단, **2026-05-31 이전 공개 가중치**) |
| 앙상블 | **금지** (단일 모델; 데이터 분할 파인튜닝 조합도 금지) |
| 허용 기법 | LoRA, Quantization, CoT, Multi-turn, **TTA** (추론 시간 내) |
| 데이터 누수 | 테스트 데이터 분석을 학습/설계에 활용 금지 (§7 참고) |
| 모델 크기 | 코드+가중치 총 80GB 이내 |
| 외부 API | 학습/추론 금지, 전처리만 3만 원 한도 |

### 본선 배점 (보고서 대비)
모델 성능+전략 40 / 데이터 활용 15 / 모델 설계 15 / 최적화 10 / 자원 효율 10 / 구축 비용 10
→ **성능만이 아니라 "왜 이 설계인가"의 논리와 효율성 문서화가 총점의 절반 이상**

## 2. 데이터 점검 결과 (EDA 완료)

### 기본 구조
- train 9,535 샘플 / test 819 샘플 (public LB는 그중 70% ≈ 573개 → **LB 노이즈 큼, 로컬 CV 신뢰 필수**)
- 각 샘플: 폴더에 4장 JPG (파일명은 순서 정보 없는 랜덤 3자리 코드, 알파벳순 = Input_1~4)
- 원본: **ActivityNet Captions** (유튜브 활동 비디오, 평균 3분, 문단 캡션이 시간순 서술)

### 발견 1 — 정답 분포의 구조
- `No_ordering=True`: 1,478개 (15.5%) → 정답 항상 `[1,2,3,4]`
- `No_ordering=False`: 8,057개 → **정답이 identity인 경우 0건**, 나머지 23개 순열에 거의 균등 분포 (~350개씩)
- test에는 `No_ordering` 컬럼이 없음 → 모델이 identity 포함 24개 순열을 모두 예측 가능해야 함

### 발견 2 — 캡션의 성격
- 원본 ActivityNet 문단을 그대로 쓰지 않고 **4프레임 시퀀스를 직접 묘사하도록 재생성된 단일 문장** (평균 24단어)
- "~begins, then~, followed by~, finally~" 등 **시간 접속사가 풍부** → 절(clause) 단위로 프레임과 정렬 가능
- 쉬운 샘플: 장면이 뚜렷이 바뀌고 캡션 절과 1:1 대응
- 어려운 샘플: 4프레임이 같은 장면의 근접 프레임(near-duplicate), 캡션이 "카메라 줌인, 손이 위로" 같은 **미세한 시각 변화**만 서술 → 고해상도 시각 비교 + 캡션의 방향성 단서(줌인/줌아웃, 이동 방향)가 결정적

### 발견 3 — ⚠️ test의 분포 이동 (약 29%가 합성 이미지)
- test 819개 중 **241개(29.4%)가 AI 생성 이미지** (1024×1536 등 고해상도, 판타지/일러스트풍)
- 이들의 캡션은 평균 76단어의 장문 서사이며 "begins by → then → after which → finally" 구조로 4절이 4프레임에 명확 대응
- **train에는 합성 샘플 0개** → 순수 OOD 일반화 테스트. 대회 introduction의 "일반화 성능" 강조 및 "외부 데이터셋 기반 추가 평가 가능" 조항과 부합
- **설계 시사점**: train 도메인(실사 유튜브 프레임)에 과적합하면 test의 30%를 통째로 잃을 수 있음 → 사전학습 VLM의 일반 능력을 보존하는 학습(LoRA 저강도, 과한 epoch 금지)이 필수
- **규칙 유의**: 이 관찰을 노골적으로 활용한 설계(예: 합성 탐지 라우터)는 규칙 3.4(테스트 특성 분석 활용 금지) 위반 소지 → §7 참고

### 발견 4 — 이미지 해상도
- train: 다양한 해상도 (176×144 ~ 640×360), test 실사분도 유사 + 합성분은 1024×1536
- 전처리에서 종횡비 보존 리사이즈 필요, 저해상도 프레임 대비 필요

## 3. 논문 3편의 시사점

### ① Dense-Captioning Events in Videos (ActivityNet Captions, 1705.00754)
- 캡션은 **시간순으로 서술된 문단**이며 비디오의 94.6%를 커버, 세그먼트 90%는 비중첩
- 대명사 co-reference("A man... → He then...")와 담화 표지("then", "finally")가 순서 단서
- 도입 장면(인트로/카메라 인사)과 종결 장면(박수/크레딧)의 시각 패턴이 정형화 → 첫/마지막 프레임 판별이 상대적으로 쉬움

### ② MERLOT (2106.02636, NeurIPS 2021)
- 프레임 순서 예측을 **pairwise 이진 분류(C(4,2)=6쌍) → 24개 순열의 pairwise log-prob 합으로 디코딩**
- 프레임 위치는 **순서 미상(unk) 임베딩**으로 인코딩 (입력 순서 누출 방지 — V1에서 위치 임베딩 누수 사고 있었음)
- 캡션-프레임 before/same/after 관계를 함께 학습하면 정렬 성능 추가 향상
- 입력 순서 랜덤 2회 TTA로 pairwise 확률 평균 (단일 모델이므로 앙상블 금지에 저촉 안 됨)
- **교훈: 순서 사전학습 재현보다, 강한 사전학습 VLM 위에 9.5k 라벨로 직접 파인튜닝이 효율적**

### ③ NACON (ACM MM 2022, 3503161.3548184) — 과제와 동일한 문제 설정
- "순서 있는 캡션 → 순서 없는 이미지 정렬"을 **order-invariant 인코더 + 크로스모달 위치 어텐션(CMPA) + 비자기회귀 포인터 디코더**로 해결
- **Exclusive loss**: 4×4 포인터 행렬에 행방향(위치→프레임) + 열방향(프레임→위치) CE 동시 적용 → exact match 직결
- 추론은 greedy 매칭 (n=4이므로 Hungarian 완전해도 비용 0)
- 어블레이션 핵심: **크로스모달(캡션) 정보가 가장 큰 단일 성능 요인** — 프레임만으로는 순서 복원이 극히 어려움
- 백본(ResNet+GloVe)을 현대 VLM으로 교체하면 구조 그대로 호환

## 4. 전략: 문제 정식화와 모델 후보

### 문제 정식화 (모든 트랙 공통)
- **24-way 순열 분류보다 구조적 출력 선호**:
  - (a) pairwise 6쌍 이진 → 24순열 스코어링 (MERLOT 방식)
  - (b) 4×4 포인터 행렬 + exclusive loss + Hungarian (NACON 방식)
  - (c) VLM 생성 출력을 24개 순열 시퀀스에 대한 **제약 디코딩/우도 스코어링**으로 채점
- 학습 시 **입력 프레임 순서 랜덤 셔플 augmentation** (라벨 재계산) → 사실상 최대 24배 증강 + 순서 불변성 확보
- 추론 시 입력 순서 TTA (2~4회 평균)

### 트랙 A (메인): 오픈소스 VLM + LoRA 파인튜닝
- **후보**: Qwen2.5-VL-7B-Instruct (1순위, 멀티이미지 네이티브, 2025-01 공개로 규정 충족), InternVL2.5-8B, LLaVA-OneVision-7B
- 4프레임 + 캡션을 넣고 순서를 출력하도록 LoRA 파인튜닝
- 출력 헤드 선택지: ① 텍스트 생성("2,4,3,1") + 24순열 우도 스코어링 ② hidden state 위에 pairwise/포인터 헤드 부착
- bf16 7B ≈ 15GB → 3090 24GB에서 추론 여유. 필요시 4bit 양자화(허용됨)
- **OOD 보존**: LoRA rank/alpha 보수적으로, 낮은 lr(~1e-5, MERLOT 레시피), early stopping, 사전학습 능력 파괴 방지

### 트랙 B (경량 대안 + 자원효율 점수용): Frozen 인코더 + NACON식 경량 헤드
- SigLIP2/CLIP 프레임 임베딩 + 텍스트 임베딩 (frozen) → 경량 크로스모달 트랜스포머 + 4×4 포인터 + exclusive loss
- 학습·추론 모두 수 분 내, 자원 효율/구축 비용 배점(20점)에 강력한 스토리
- 트랙 A와 성능 비교 후 최종 1개 선택 (앙상블 금지이므로 결합 불가)

### 트랙 0 (즉시 실행 베이스라인, 파인튜닝 없음)
1. **항상 `[1,2,3,4]`** → LB에서 identity 비율 확인 (train 기준 ~15% 예상)
2. **CLIP/SigLIP zero-shot 절-프레임 정렬**: 캡션을 절 단위 분할 → 프레임×절 유사도 행렬 → Hungarian
3. **Qwen2.5-VL zero-shot 프롬프팅** (CoT 허용)
→ 셋 다 제출해 LB 감각 확보 + test 분포(identity 비율 등) 간접 파악

### 검증 설계
- train 9,535 → 90/10 분할 (Answer 순열 stratified, No_ordering 비율 유지)
- 지표: Exact Match (주지표) + pairwise accuracy·Kendall τ (진단용)
- 합성 도메인 홀드아웃이 불가능(train에 0개)하므로, **CV-LB 갭 자체를 OOD 일반화 신호로 모니터링**

## 6. 성능 목표 (가설)

| 단계 | Exact Match 예상 |
|---|---|
| identity 고정 | ~15% |
| CLIP zero-shot 정렬 | 20~35% |
| VLM zero-shot | 30~50% |
| VLM LoRA 파인튜닝 | 55~75%+ (실사분), 합성분은 캡션 명확성 덕에 그 이상 가능 |

*합성 30%는 캡션-프레임 대응이 명확해 오히려 고득점 구간일 수 있음. 실사 near-duplicate 케이스가 최종 병목이 될 것.*

## 7. 규칙 준수 체크리스트 (실격 리스크 관리)

1. **original_data(ActivityNet) 학습 사용 금지** — 외부 데이터. 데이터 구성 이해 용도로만. 특히 **test 캡션을 원본과 매칭해 순서를 복원하는 행위는 명백한 누수+외부데이터 위반** → 절대 금지
2. **테스트 특성 분석의 활용 경계 (규칙 3.4)**: 합성 이미지 발견 사실을 학습 전처리·모델 설계에 직접 반영(예: 합성 전용 분기) 금지. "일반화 성능 보존" 같은 도메인 불문 원칙으로만 설계. 보고서에도 test 분석 기반 설계로 읽힐 서술 배제
3. **앙상블 금지**: 트랙 A/B 중 하나만 최종 선택. TTA(동일 모델, 입력 변형 평균)는 허용 범위로 판단되나 애매하면 Discussion 탭에 사전 문의
4. **모델 가중치 공개일**: 사용 전 HuggingFace 공개일 2026-05-31 이전 확인 + 라이선스 확인
5. **제출 전 3090 1장 환경 검증** (현재 로컬에 3090 2장 — 학습은 2장 활용 가능, 추론은 반드시 1장으로 테스트)
6. 팀/계정 규정: 다계정 금지, 팀 결성 후 변경 불가
